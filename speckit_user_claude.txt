  # Auto-Commit Changes

  Automatically stage and commit all changes after a Spec Kit command completes.

  ## Behavior

  This command is invoked as a hook after (or before) core commands. It:

  1. Determines the event name from the hook context (e.g., if invoked as an `after_specify` hook,
  the event is `after_specify`; if `before_plan`, the event is `before_plan`)
  2. Checks `.specify/extensions/git/git-config.yml` for the `auto_commit` section
  3. Looks up the specific event key to see if auto-commit is enabled
  4. Falls back to `auto_commit.default` if no event-specific key exists
  5. Uses the per-command `message` if configured, otherwise a default message
  6. If enabled and there are uncommitted changes, runs `git add .` + `git commit`

  ## Execution

  Determine the event name from the hook that triggered this command, then run the script:

  - **Bash**: `.specify/extensions/git/scripts/bash/auto-commit.sh <event_name>`
  - **PowerShell**: `.specify/extensions/git/scripts/powershell/auto-commit.ps1 <event_name>`

  Replace `<event_name>` with the actual hook event (e.g., `after_specify`, `before_plan`,
  `after_implement`).

  ## Configuration

  In `.specify/extensions/git/git-config.yml`:

  ```yaml
  auto_commit:
  default: false # Global toggle — set true to enable for all commands
  after_specify:
  enabled: true # Override per-command
  message: "[Spec Kit] Add specification"
  after_plan:
  enabled: false
  message: "[Spec Kit] Add implementation plan"
  ```

  ## Graceful Degradation

  - If Git is not available or the current directory is not a repository: skips with a warning
  - If no config file exists: skips (disabled by default)
  - If no changes to commit: skips with a message

  ## Behavior

  This command is invoked as a hook after (or before) core commands. It:

  1. Determines the event name from the hook context (e.g., if invoked as an `after_specify` hook,

----------------------------------------------------------------------------------------------------
  Four reasons that, taken together, made SQLite + R-tree the path of least resistance — and three
  alternatives that each failed on at least one of them.

  ### What the index actually has to do

  | Need | Source |
  |---|---|
  | Persist tile metadata across process restarts | FR-017 |
  | Selectively UPDATE a single row when one tile's `(mtime, size, hash)` no longer matches disk |
  FR-018 |
  | Answer "which tiles' bboxes contain (lat, lon)?" cheaply, for every point and every profile
  sample | FR-014, hot path for SC-001/002 |
  | Reload in < 5 s for a fresh process; survive a crash mid-registration | SC-009; 5-minute
  registration window means crashes during it are realistic |
  | Not introduce a new runtime dependency | Constitution Principle III — new deps must clear a
  justification bar |

  ### Why SQLite cleared all five

  1. **Stdlib, zero new dependency.** Python ships `sqlite3` and CPython's standard build includes
  the R-tree extension. Under Principle III's "new dependency must be justified in writing" bar,
  that's the cheapest possible answer. `pyarrow`, `duckdb`, `structlog`, `lmdb` would each cost a
  justification entry.
  2. **Atomic single-row UPDATE is what FR-018 literally asks for.** When the staleness check finds
  one tile rewritten, we want to refresh that row and leave the other 1,755 alone. SQLite does this
  in one statement inside a transaction. A JSON / CSV / Parquet sidecar would force a whole-file
  rewrite for every stale tile, multiplying disk IO and creating a corruption window on every
  write.
  3. **R-tree virtual table makes the spatial lookup an indexed query for free.** `SELECT path FROM
  tiles_rtree WHERE :lat BETWEEN min_lat AND max_lat AND :lon BETWEEN min_lon AND max_lon` runs in
  O(log N). At 1,756 rows even a linear scan would be sub-millisecond, but R-tree means we don't
  have to think about it again if the dataset grows 10× or 100×. No implementation cost on our
  side.
  4. **WAL mode is crash-safe.** Registration takes up to 5 minutes (SC-008). A process kill or OOM
  4 minutes in is realistic, and we don't want that to corrupt prior work. SQLite's write-ahead log
  gives transactional durability essentially for free.

  ### Why I rejected the alternatives

  | Alternative | Failed on |
  |---|---|
  | **JSON sidecar** | No atomic single-row update; full read-modify-write per FR-018 refresh; no
  indexed lookup; load entire file to memory on every open |
  | **CSV / TSV** | Same as JSON, plus encoding hassle |
  | **Parquet** | Columnar scans are great, but updates rewrite the whole file. Plus a pyarrow
  runtime dep — fails Principle III's bar |
  | **DuckDB embedded** | Powerful query engine for a workload that's not analytical. Extra dep
  without a matching benefit |
  | **LMDB / BerkeleyDB** | Fast KV, but no spatial indexing, harder to inspect, extra dep |
  | **Pickle on shutdown** | Loses everything on crash; no atomic updates; no indexing |

  ### The bonus property I wasn't required to deliver but get anyway

  The sidecar is inspectable from the command line: `sqlite3 .us_elevation_profile_index.sqlite
  ".schema"` and ad-hoc SELECTs work without any tooling. For a library a user might reach for to
  debug coverage issues, that's a useful affordance — and the cost was zero.

  If the index ever grew past, say, 100 GB, or the workload became analytical (aggregating across
  many tiles), I'd revisit — but for a few thousand rows of metadata serving point-and-path
  lookups, SQLite is the boringly-correct choice.

  Good question — and it's worth separating two things "Zarr instead" could mean, because the
  answer is different for each.

  ### Possibility 1: Zarr replacing the SQLite **index**

  This one's quick: wrong tool category. Zarr is an N-D array store; the index is a relational
  catalog of per-tile metadata (path, bbox, resolution, mtime, hash). What we need from the index
  is:

  - Indexed bbox lookup → SQLite R-tree does this; Zarr doesn't have an equivalent.
  - Atomic single-row UPDATE on stale-tile refresh (FR-018) → SQLite transactions; Zarr has no row
  semantics.
  - Crash safety mid-registration → SQLite WAL; Zarr writes are chunk-granular but the index isn't
  an array.

  Zarr's `.zattrs` is JSON metadata, not a queryable catalog. So this scenario doesn't really make
  sense.

  ### Possibility 2: Zarr replacing GeoTIFF as the **tile data** store

  This is the more interesting reading. The idea would be: on registration, transcode the user's
  ~1,756 GeoTIFFs into a single Zarr store and read chunks from there instead of doing rasterio
  windowed reads on the originals. I considered it. Four reasons it's the wrong call **for this
  product**:

  **1. Rasterio already gives us what Zarr would give us.** Both formats support lazy chunk/window
  reads at the cost of one decode per access. Both compress on disk. There is no missing capability
  that Zarr would unlock — the windowed-read pattern in R-02 / R-06 is functionally equivalent to
  Zarr chunk fetches. So you're paying a transcode cost for no new capability.

  **2. The transcode cost blows past SC-008.** SC-008 budgets 5 minutes for cold registration. At
  23 billion 64-bit samples (184 GB raw), reading + decompressing + recompressing + writing through
  Zarr takes 10–30 minutes realistically. You'd either miss the budget or have to declare a
  separate one-time "first-time setup" phase outside the spec.

  **3. Heterogeneous grids make a single Zarr array fictional.** CONUS NED, Hawaii NED, and Puerto
  Rico NED aren't on a common projection or resolution. To stitch them into *one* logical Zarr
  array you'd either:
  - reproject everything to a common grid (lossy, slow, alters the source data — a contract
  violation under SC-003 "values agree with the source dataset"), or
  - keep multiple Zarr arrays (one per region/resolution), which just hands us back per-tile lookup
  — exactly what rasterio + the SQLite index already does.

  **4. The user said "store the files" and v1 explicitly excludes a decoded-payload cache.**
